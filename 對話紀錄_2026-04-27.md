## 一、起因:pH 校準誤用了 SEN0161-v2 程式

實際感測器是 **SEN0169**,但誤用了 [校準程式/PH_SEN0161v2/PH.ino](校準程式/PH_SEN0161v2/PH.ino)。
該程式的 `DFRobot_PH` 函式庫會把校準電壓寫進 UNO EEPROM(位址 0x00 / 0x04),公式假設又跟 SEN0169 完全不同 → 校準值污染。

### 處理
1. 新增 [校準程式/PH_EEPROM_Reset/PH_EEPROM_Reset.ino](校準程式/PH_EEPROM_Reset/PH_EEPROM_Reset.ino)
   把 EEPROM 還原成 DFRobot 預設值(1500 mV / 2032.44 mV)
2. 燒錄一次,確認 BEFORE/AFTER 值

## 二、改用兩點軟體校準(SEN0169 專屬)

新增 [校準程式/PH_TwoPoint/PH_TwoPoint.ino](校準程式/PH_TwoPoint/PH_TwoPoint.ino),不依賴硬體 gain pot,純軟體解 slope+offset。

### 流程坑點
- 第一次校準 V7=1.9047, V4=1.6057 → ΔV 只有 0.299 V,**slope 高達 10.03,訊號太弱**
- 邊看序列埠邊轉藍色 gain 電位器,把 V4 拉到 ~1.0 V 讓 ΔV 接近 0.85 V
- 第二次校準成功:**slope=3.5686, offset=0.3519**(slope 接近理論值 3.5)

### 套用至主程式
[uno_sensor_serial/uno_sensor_serial.ino:99](uno_sensor_serial/uno_sensor_serial.ino#L99):
```cpp
v["ph"] = 3.5686 * phVoltage + 0.3519;
```

### 重要警語
**藍色 gain 電位器位置已固定,不可再轉動**,否則整組校準作廢需重做。建議貼透明膠帶封住。

## 三、TDS 校準(踩坑兩次)

### 第一次(錯誤)
用 **12.88 mS/cm EC 校準液**,輸入 `cal:707`(誤以為這瓶是 707 ppm)
→ 真實該瓶 ≈ 6440 ppm,且超出 SEN0244 量程(0~1000 ppm)
→ 算出 K=0.58 完全是垃圾值

### 第二次(正確)
換成 **707 ppm TDS 標準液**,輸入 `cal:707`
→ 得到 **K=0.60**(已套到主程式 [uno_sensor_serial.ino:103](uno_sensor_serial/uno_sensor_serial.ino#L103))

### 重要警語
**海水真實 TDS 約 35000 ppm,SEN0244 量程只到 1000 ppm**,在海水中讀值會嚴重飽和。即使有校準,海水 TDS 數字**只能當趨勢看,不能視為真實 ppm**。

## 四、EC 校準

### 確認資訊
- 感測器型號:**DFR0300-H**(K=10,高量程 10~100 mS/cm)
- 環境:**海水**(EC ~35~55 mS/cm)
- 校準液:**12.88 mS/cm**
- 結果:**K=0.85**

> Word 檔 sensor.docx 表格寫「EC (K=1)」是舊資料,**實際使用者已換成 K=10 版本**,以實際為準。

### 套用至主程式
[uno_sensor_serial.ino:111-120](uno_sensor_serial/uno_sensor_serial.ino#L111-L120) 新增完整 DFR0300-H 公式(主程式之前根本沒有 EC 換算!):
```cpp
const float EC_K = 0.85f;
float ecVoltage_mV = ecAvgADC * (5000.0 / 1023.0);
float ecRaw = ecVoltage_mV / 164.0;  // 164 = RES2(820) * ECREF(200) / 1000
float ecTempCoeff = 1.0 + 0.0185 * (currentTemp - 25.0);
float ecValue = ecRaw * EC_K / ecTempCoeff;
v["ec"] = ecValue;  // 單位 mS/cm (改為 float,之前是 int ADC)
```

## 五、濁度(SEN0189)

主程式之前**只輸出原始電壓**,沒有換算 NTU。改用 DFRobot 官方公式:
```
V > 4.2V       → NTU = 0
V < 2.5V       → NTU = 3000
其他           → NTU = -1120.4*V² + 5742.3*V - 4352.9
```

[uno_sensor_serial.ino:122-131](uno_sensor_serial/uno_sensor_serial.ino#L122-L131)

### 燒錄前必檢查
模組上的 A/D 切換開關必須在 **A(類比)** 位置,否則只能讀到 0V 或 5V。

## 六、CO2 校準(MH-Z19B / MH-Z19C)

### 設計目的
藻類池是密閉低 CO2 環境,**ABC 自動校準會把長期低點誤認為 400 ppm**,導致讀值漂移。
解法:關閉 ABC + 在戶外做手動歸零。

### 校準程式
[校準程式/CO2_MHZ19/CO2_MHZ19.ino](校準程式/CO2_MHZ19/CO2_MHZ19.ino) 支援兩種模式:

**(A) 自動模式**(行動電源 + 戶外):
1. 燒錄 → 接行動電源 → 帶到戶外
2. **L LED 慢閃 25 分鐘**(暖機倒數)
3. **L LED 恆亮** = 自動執行 abcoff + zero 完成
4. 帶回燒主程式

**(B) 手動模式**(電腦序列埠):
- 指令:`abcoff` → `zerob` / `zeroc` → `read`

### 校準現況
- **C 感測器**:已校準(2026-04-27),但 LED **5 分鐘就恆亮**(預期 25 分鐘),
  可能是行動電源啟動電壓不穩或 EMI 干擾觸發 skip。
  序列埠驗證讀值 **500 ppm 室內**(合理範圍),哈氣測試正常。
  **校準應該成功,但時間異常待後續觀察**。
- **B 感測器**:**未校準**,接上後再做一次自動模式即可。

### 待改進(未做)
- 加開機後 30 秒延遲再開始倒數(避免電壓不穩誤觸發)
- 校準完用 LED 快閃 5 次再恆亮,區分「正常完成」vs「異常恆亮」

2. **Payload 把 `t` 改成 `temp`** ([rpi_gui_monitor.py:137](rpi_gui_monitor.py#L137)),
   Apps Script 端必須用 `temp` 接收溫度。
3. **失敗訊息只印到 terminal**,GUI 看不到。建議從 terminal 啟動以便除錯:
   ```bash
   python3 rpi_gui_monitor.py
   ```
4. **校準後 EC 和 turb 數字大小改變很多**:
   - EC:`512`(ADC 整數) → `42.35`(mS/cm 浮點)
   - turb:`2.5`(電壓) → `856`(NTU 浮點)
   雲端上傳結構不變,只是欄位數字看起來會差很多。

### 建議刪除/封存
